## MOBILE-LITE Render Mode — Implementation Log

### 2026-06-05 23:45+02:00 — Design & first pass
**Objective:** Implement a parallel "lite" render branch for mobile / low-power GPUs. Desktop path must remain byte-identical.

**Files touched:**
- `utils/deviceCapabilities.ts` (new)
- `utils/geometryConstants.ts`
- `utils/shaderRegistry.ts`
- `utils/shaderVersion.ts`
- `hooks/useWebGPURender.ts`
- `components/PatternDisplay.tsx`
- `App.tsx`

**Key decisions:**
- Capability detection is conservative: mobile UA string, missing WebGPU, and optional async low-power GPU adapter hint. URL param `?lite=1|0` and localStorage `xasm1_lite_mode` override auto-detection for desktop testing.
- Centralized policy object `DEVICE_CAPABILITIES` exported from `deviceCapabilities.ts`; consumers read flags, they don't re-detect.
- `patternv0.21.wgsl` tagged as `liteRecommended` in `shaderRegistry.ts` — horizontal layout, single-pass composite, no alpha blending, no layered bloom, supports 32-step paging.
- Lite mode auto-substitutes the display shader via `displayShaderFile` in `App.tsx` without mutating the user's stored desktop preference. Toggling lite off restores the previous shader instantly.
- Bloom initialization is skipped entirely in lite mode (`PatternDisplay.tsx` effect returns early when `liteMode` is true).
- WebGL overlay is disabled in lite mode (`isOverlayActive = !liteMode && WEBGL_HYBRID_SHADERS.has(...)`), saving an extra context.
- Canvas metrics reduced to 512×512 in lite mode (half the desktop 1024×1024), cutting fill-rate and memory.
- Compute path gated by `liteMode` in `useWebGPURender.ts` — falls back to CPU packing on the unlikely event a high-precision shader is forced in lite mode.
- DPR capped to 1.0 in lite mode (`PatternDisplay.tsx` `syncCanvasSize`), cutting physical canvas buffer size by up to 4× on high-DPR mobile screens.
- Desktop path is untouched: when `liteMode` is false, every code branch is identical to pre-change behavior (verified by conditional checks).

**Status:**
- `npm install` ✅ (missing `@tanstack/react-query` and `zod` installed)
- `npm run typecheck` ✅ (zero errors)
- `npm run build` ✅ (4 GB heap, 18.3s)

  1. `?lite=1` forces lite mode, substitutes v0.21, skips bloom, disables overlay, renders at 512×512.
  2. `?lite=0` forces full desktop mode even on mobile UA.
  3. Desktop default (`?lite` absent, localStorage empty, desktop UA) renders exactly as before (same shader, same bloom, same canvas size).
  4. No WebGPU console errors in either path.

## Parser → Renderer Pipeline Hardening — Implementation Log

### 2026-06-19 18:02+02:00 — Plan approved

**Files touched:**

**Key decisions:**
- Extracted a testable `createParserPromise` helper in `utils/parserPromise.ts` that owns message/error/messageerror/timeout listener lifecycle and terminates the worker on timeout.
- `parserWorker.ts` now delegates to `createParserPromise`; `parseInWorker` no longer leaks listeners on any path.
- Worker now posts a `'fetch'` progress stage and aborts the CDN libopenmpt fetch after 12 s, so a stalled CDN request fails before the outer 15 s parser timeout.
- `useLibOpenMPT.ts` no longer gates the worker on the HEAD check (Vite hashes worker chunks in production, so the source `.ts` URL may 404 even though the emitted chunk is valid). The real availability test is `new Worker()` + error handling.
- Worker creation is wrapped in `try/catch`; constructor errors (404, CSP, MIME) immediately mark the worker unhealthy and fall through to main-thread parse.
- `resolveParsedModule` terminates and discards the worker in a `finally` block after any worker failure/timeout, guaranteeing the next load gets a fresh worker.
- Status machine now shows `(fetching parser engine)` / `(reading patterns)` progress and falls back silently to main-thread parse when the worker path fails.
- WebGPU fallback logs a clear `[Renderer] WebGPU unavailable (<reason>); falling back to <backend>` line exactly once per session; `PatternDisplay.tsx` guards against re-applying the fallback.
- Added `hasWebGPUAutoFallbackApplied()` helper for the session guard.
- `?renderer=webgpu` still attempts WebGPU first; fallback only on real init failure.
