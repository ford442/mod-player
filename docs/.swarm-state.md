### Root Cause Summary
The channel-count mismatch was **already fixed in HEAD**. The packer and GPU buffer allocator both derive `numChannels` from `rawChannels + (padTopChannel ? 1 : 0)`, and the shader receives the same `numChannels` via uniforms. The historical "320 cells into 256-cell buffer" log no longer reproduces. The only remaining gap was the absence of a runtime invariant that would catch a future regression.

## Iteration 2 — Amber-vs-Blue Emitter Differentiation — 2026-04-24T16:03:00+02:00

### Status
Complete

### Files changed
- `shaders/patternv0.50.wgsl`

### What I found
- The v0.50 shader already had three emitters and DURA sustain logic, but the top emitter was always blue (via `calculateBlueIntensity`) and the bottom was always amber for expression-only steps.
- The shader constants `NOTE_MAX = 96u`, `NOTE_OFF = 97u`, etc. did not match the packer's `NOTE_MAX = 119` / `NOTE_OFF_MIN = 120`. This was a shader-side bug that would have misclassified notes 97–119.
- The data path uses a `storage` buffer (`cells: array<u32>`) with flat varyings, not an `RG32UI` texture as described in the task brief. The actual bit layout in `packPatternMatrixHighPrecision` is:
  - packedA: `[note:8 | inst:8 | duration:8 | volPacked:8]`
  - packedB: `[effCmd:8 | effVal:8 | durationFlags:7 | reserved:1 | volCmd:8]`
- The packer already pre-computes the expression-only flag in bit 7 of `inst`, so the shader can use `isExpressionOnly` directly without reconstructing it from volCmd/effCmd.

### What changed
1. **Fixed note-range constants** to match packer: `NOTE_MAX = 119u`, `NOTE_OFF_MIN = 120u`.
2. **Replaced dead functions** `isSustaining` and `calculateBlueIntensity` with a new `calculateTopIntensity` that branches on `isNoteOn / isExprOnly / isSustain`.
3. **Reclassified cell types** in the fragment shader:
   - `isNoteOn`: note > 0 && note < 120 && rowOffset == 0
   - `isNoteOff`: note >= 120
   - `isExprOnly`: !isNoteOn && !isNoteOff && isExpressionOnly
   - `isSustain`: note > 0 && note < 120 && duration > 1 && rowOffset > 0 && !isNoteOff
   - `isDead`: everything else
4. **Rerouted top emitter**: blue (`#0080FF`) for note-on, amber (`#FFA500`) at ~0.7 intensity for expression-only, very faint blue (0.1) for sustain, dark for dead/note-off.
5. **Rerouted middle emitter**: pitch-class color at full intensity for note-on, ~45 % fixed intensity for sustain, dark (`#0D0D0D`) for expression-only and dead.
6. **Rerouted bottom emitter**: suppressed for expression-only (amber moved to top); only subtle amber when note-on/sustain rows also carry expression data.
7. **Updated external glow** to use new `isNoteOn / isSustain` booleans.

## PER-INSTRUMENT PALETTE — Implementation Log

### Iteration 1 — 2026-06-12T15:32:31+02:00

#### Goal
Implement a user-toggled per-instrument color palette mode for the WebGPU pattern visualizer, default-off, with the existing pitch-hue path unchanged.

#### Files touched
- `types.ts` — Added `LibOpenMPT` bindings for `_openmpt_module_get_num_instruments` and `_openmpt_module_get_instrument_name`; extended `WorkerParseMetadata` with `numInstruments` and `instruments`.
- `workers/openmpt-parser.worker.ts` — Extracts instrument count and names from the worker-side libopenmpt module and returns them in parse metadata.
- `hooks/useLibOpenMPT.ts` — Added `instrumentNames` state, populated from worker metadata, returned to consumers; cleared on module destroy.
- `utils/instrumentPalette.ts` — New deterministic palette generator using golden-ratio hue rotation plus name hashing; outputs `Uint8Array` sized for a 64-entry 1-D rgba8unorm texture.
- `utils/gpuPacking.ts` — Extended uniform payload with `paletteMode` at slot `[32]`; extended return byte length is now 132.
- `hooks/useWebGPURender.ts` —
  - Added `paletteMode` / `instrumentPalette` to `WebGPURenderParams`.
  - Uniform buffer data size bumped to 144 bytes; extended uniform buffer allocation bumped to 132 bytes.
  - Creates a placeholder 1-D palette texture for `v0.56`, updates it when `instrumentPalette` changes, and binds it at `@group(0) @binding(7)`.
  - Added `v0.56` to all version-detection chains (`isHighPrec`, cell sizing, `playheadRowAsFloat`, debug overlay).
- `components/PatternDisplay.tsx` — Added `paletteMode`/`instrumentPalette` props and render-params; added `v0.56` to `padTopChannel` and `canvasMetrics` version chains.
- `shaders/patternv0.56.wgsl` + `public/shaders/patternv0.56.wgsl` — New fork of `v0.51` with `paletteMode` uniform, `@binding(7) texture_1d<f32> instrumentPalette`, and per-instrument color lookup in the middle emitter.
- `utils/shaderRegistry.ts` — Registered `patternv0.56.wgsl` with circular LED metadata.
- `appConfig.ts` — Added `v0.56` to `SHADER_GROUPS.CIRCULAR`.
- `src/utils/shaderHelpers.ts` — Added `v0.56` to trap-circular category and description.
- `App.tsx` —
  - Added persisted `paletteMode` state (`xasm1_paletteMode`).
  - Computes `instrumentPalette` from `instrumentNames` with `useMemo`.
  - Feeds `instruments` into `moduleMetadata`.
  - Resets `paletteMode` to pitch-hue when leaving `v0.56`.
- `components/MainLayout.tsx` — Added palette-mode props and a `By Pitch / By Instrument` toggle next to the existing palette selector; passes data through to `PatternDisplay`.

#### Key design decisions
1. **1-D texture path chosen** for the palette (not uniform array) to match the existing `texture_1d<f32>` pattern used by `v0.55` and to avoid uniform size limits.
2. **Palette slot fixed at 64 entries** with shader wrapping via `inst % 64u`; modules with more instruments wrap deterministically.
3. **`paletteMode` placed at uniform slot `[32]`** (byte offset 128) so it does not collide with existing extended-layout fields (`innerRadius`, `outerRadius`, night-mode fields, etc.). The `v0.56` WGSL `Uniforms` struct mirrors slots 0–31 as explicit padding before `paletteMode` to guarantee byte alignment.
4. **Default path preservation:** With `paletteMode` defaulting to `0`, `fillUniformPayload` writes a zero into slot `[32]`; existing shaders only read up to slot `[24]` (colorPalette) or up to slot `[31]` (night-mode shaders), so their consumed bytes are unchanged.
